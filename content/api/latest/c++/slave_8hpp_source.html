488  // TODO(qianzhang): This is a workaround to make the default executor task's
489  // volume directory visible in MESOS UI. It handles two cases:
490  // 1. The task has disk resources specified. In this case any disk resources
491  // specified for the task are mounted on the top level container since
492  // currently all resources of nested containers are merged in the top
493  // level executor container. To make sure the task can access any volumes
494  // specified in its disk resources from its sandbox, a workaround was
495  // introduced to the default executor in MESOS-7225, i.e., adding a
496  // `SANDBOX_PATH` volume with type `PARENT` to the corresponding nested
497  // container. This volume gets translated into a bind mount in the nested
498  // container's mount namespace, which is not visible in Mesos UI because
499  // it operates in the host namespace. See MESOS-8279 for details.
500  // 2. The executor has disk resources specified and the task's ContainerInfo
501  // has a `SANDBOX_PATH` volume with type `PARENT` specified to share the
502  // executor's disk volume. Similar to the first case, this `SANDBOX_PATH`
503  // volume gets translated into a bind mount which is not visible in Mesos
504  // UI. See MESOS-8565 for details.
505  //
506  // To make the task's volume directory visible in Mesos UI, here we attach the
507  // executor's volume directory to it so that it can be accessed via the /files
508  // endpoint. So when users browse task's volume directory in Mesos UI, what
509  // they actually browse is the executor's volume directory. Note when calling
510  // `Files::attach()`, the third argument `authorized` is not specified because
511  // it is already specified when we do the attach for the executor's sandbox
512  // and it also applies to the executor's tasks. Note that for the second case
513  // we can not do the attach when the task's ContainerInfo has a `SANDBOX_PATH`
514  // volume with type `PARENT` but the executor has NO disk resources, because
515  // in such case the attach will fail due to the executor's volume directory
516  // not existing which will actually be created by the `volume/sandbox_path`
517  // isolator when launching the nested container. But if the executor has disk
518  // resources, then we will not have this issue since the executor's volume
519  // directory will be created by the `filesystem/linux` isolator when launching
